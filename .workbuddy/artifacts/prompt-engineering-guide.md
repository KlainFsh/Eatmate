# 智食 Qwen 提示词工程指南

> 目的: 不断迭代 Prompt，找到最准确、最稳定的食物识别效果
> 原则: 每一次修改都要有明确理由，不要堆砌无意义的修饰词

---

## 一、Prompt 的正确心态

**Qwen 不是搜索引擎，你是在教一个聪明但没见过你照片的实习生。**

它需要：
1. 知道自己是谁（角色）
2. 知道你要什么格式（结构约束）
3. 知道什么是"对"的（判断标准）
4. 知道不确定时怎么办（兜底规则）

---


## 二、逐级优化过程（边测边改）

### Level 1 — 最简版（基线，先跑一下看看错在哪）

```
识别这张照片里有什么食物，输出 JSON：{"dishes":[{"name":"","calories":0}]}
```

跑完看结果：菜名对了吗？多菜品漏了吗？热量离谱吗？**先知道弱在哪，再对症下药。**

---

### Level 2 — 加角色 + 加字段（解决数据不全）

```text
你是一位持有国家营养师认证的专业人士。分析这张食物照片，返回 JSON：

{
  "dishes": [
    {"name": "", "estimated_weight_g": 0, "calories_kcal": 0, "protein_g": 0, "carb_g": 0, "fat_g": 0}
  ],
  "total": {"calories": 0, "protein": 0, "carb": 0, "fat": 0}
}
```

**为什么加这个**:
- "国家营养师认证" = 框定知识域，让模型调用营养学知识而非通用常识
- 加 `estimated_weight_g` = 逼模型估算份量，没有重量就没有准确热量

---

### Level 3 — 加判断约束（解决瞎编问题）

```text
你是一位持有国家营养师认证的专业人士。请仔细分析这张食物照片。

规则：
- 只识别你确定看到的食材和菜品，不要猜测模糊部分
- 如果只看清一道菜，就只返回一道，不要编造
- 如果某道菜不确定，confidence 设为 0.6 以下
- 中餐份量参考：外卖一份菜约250-400g，食堂打菜约150-250g，面类约400-600g

返回严格的 JSON（不要 markdown 包裹，不要任何解释性文字）：

{
  "dishes": [
    {
      "name": "中文菜名",
      "estimated_weight_g": 数字,
      "calories_kcal": 数字,
      "protein_g": 数字,
      "carb_g": 数字,
      "fat_g": 数字,
      "confidence": 0-1之间,
      "identified_ingredients": ["可见食材"]
    }
  ],
  "total_calories": 数字,
  "total_protein": 数字,
  "total_carb": 数字,
  "total_fat": 数字
}
```

**为什么加这些**:
- "不要猜测模糊部分" = 降低幻觉
- "只返回一道" = 防止多菜品场景下编造
- "份量参考" = 给模型一个锚点，防止估算 50g 或 2000g 这种极端值
- "strict JSON" = 减少后处理成本

---

### Level 4 — 加细分场景识别（解决食堂/外卖识别差的问题）

```text
你是一位服务过500+减脂客户的国家注册营养师。请分析这张食物照片。

【先判断场景】
看餐具、环境、菜品摆盘，判断是：外卖（塑料盒/纸盒）、食堂（不锈钢餐盘/多格盘）、家常（陶瓷碗碟）、餐厅（精致摆盘）、还是其他。

【然后识别菜品】
- 外卖常见：黄焖鸡、麻辣香锅、酸菜鱼、盖浇饭、拌面、轻食沙拉
- 食堂常见：西红柿炒蛋、红烧肉、土豆丝、青菜、麻婆豆腐、米饭（通常150-200g一份）
- 注意区分相似菜品：红烧肉vs东坡肉、宫保鸡丁vs辣子鸡、麻婆豆腐vs家常豆腐

【份量估算】
- 外卖一份主菜通常250-400g，米饭约200-300g
- 食堂一格菜约150-250g，米饭约150-200g
- 家常一碗菜约200-350g，一碗米饭约150-200g
- 面类/汤类连汤带料约400-600g

【热量估算参考】
- 荤菜(炸/红烧)约200-300kcal/100g，荤菜(清蒸/白切)约120-180kcal/100g
- 素菜(炒)约80-120kcal/100g，素菜(凉拌/清蒸)约30-60kcal/100g
- 米饭约116kcal/100g，面条(煮)约110kcal/100g
- 烹饪用油每勺(10g)增加约90kcal，注意菜品表面的油光判断用油量

返回严格 JSON：
{
  "dining_scene": "外卖/食堂/家常/餐厅/其他",
  "dishes": [
    {
      "name": "",
      "estimated_weight_g": 0,
      "calories_kcal": 0,
      "protein_g": 0,
      "carb_g": 0,
      "fat_g": 0,
      "confidence": 0,
      "cooking_method": "炒/蒸/煮/炸/烤/炖/凉拌",
      "oil_level": "少油/正常/偏油",
      "identified_ingredients": []
    }
  ],
  "total": {"calories": 0, "protein": 0, "carb": 0, "fat": 0},
  "is_complete_meal": true,
  "notes": "如有不确定之处在此说明"
}
```

**为什么 Level 4 可能是最优版本**:
- **场景先置**: 先判断场景再识别，给模型一个认知框架（食堂的红烧肉和餐厅的红烧肉热量不同）
- **常见菜列举**: 给模型候选集，缩小搜索空间，提高准确率
- **份量锚点**: 分场景给份量参考，比笼统的"300g"精准得多
- **热量公式**: 给模型计算方法论，而不是让它凭空猜
- **油量判断**: 中餐热量最大的变量就是油，提示它看油光
- **cooking_method + oil_level**: 这两个标签比纯粹的数字更有解释力

### Level 5 — 终极版（如果 Level 4 还不够好，加这两招）

```text
[在上面 Level 4 的基础上追加]

【不确定时的处理】
- 如果图片模糊、暗光、遮挡严重，在最前面加一句 "⚠️ 图片质量不佳，以下为估算："
- 如果一道菜可能是A也可能是B（如分不清"宫保鸡丁"和"辣子鸡丁"），confidence 设 0.5-0.7
- 如果是混合菜（如麻辣烫、火锅），将能识别的主要食材列出来，而非强行给单一菜名

【输出格式严格要求】
返回内容必须是纯 JSON，第一个字符必须是 {，最后一个字符必须是 }。不要用 ```json 包裹，不要在前面加任何文字。
```

**为什么加**:
- "混合菜处理" = 麻辣烫是高频场景，强制单一菜名反而错
- "纯 JSON" = 反复强调，减少格式错误

## 四、常见翻车场景 & Prompt 补救

| 翻车表现 | 可能原因 | Prompt 补救 |
|----------|----------|-------------|
| 菜名英文 | 模型默认用英文 | 加"所有菜名使用中国大陆通用名称" |
| 多菜品只识别一个 | 注意力集中在中央 | 加"请扫描照片中的所有区域，不要遗漏边缘的菜品" |
| 把辣椒当主菜 | 颜色鲜艳吸引注意力 | 加"调味料（辣椒、花椒、葱花）不算作独立菜品"** |
| 汤被当菜 | 分不清液体 | 加"汤品标注为'XX汤'而非菜品" |
| 重量偏差2-3倍 | 缺少参照物 | 加"以照片中的筷子/勺子/餐盘作为大小参照" |
| 米饭被忽略 | 白色辨识度低 | 加"白色主食（米饭、馒头、花卷）也需要单独识别" |

---

## 五、最终建议

**从 Level 4 开始测。** Level 4 在信息量和 tokens 消耗之间取得了最好的平衡——它给了模型足够的"知识框架"（份量锚点、热量公式、场景区分），但没有冗长到浪费时间。

如果 Level 4 某个特定维度翻车（比如外卖场景还是不准），不要全部重写，而是**针对翻车点加一句补充规则**，单独再测那个场景。
